7 # Animation Basics
8 
9 In order to animate a digital character, one may choose from two main animation techniques:
10  - skeleton-based animation, in which the character's mesh is bound to an
11 animation skeleton composed of joints, sometimes called bones.
12  - cage-based animation, in which the character's mesh is bound to a set of
13 points defining an emglobing deformation cage.
14 
15 In both cases, the character's mesh is bound to the handles (skeleton joints or
16 cage points) through a set of skinning weights.
17 During the animation, the handles transformations are first updated with respect
18 to the animation pose, then the mesh is deformed by combining the handles
19 transformations through the skinnning weights.
20 
21 # Character Animation in Radium
22 
23 The Radium Engine provides the basic classes for character animation.
24 
25 Starting from a `Ra::Core::Animation::HandleArray`, which represents a deformation metaphor, such as `Ra::Core::Animation::Cage` or `Ra::Core::Animation::Skeleton`,
26 one must associate influence weights to each vertex of a mesh within a `Ra::Core::Animation::WeightMatrix`
27 (rows representing vertices, columns handles).
28 These influence weights, or skinning weights, are then used by a skinning algorithm to deform
29 the associated mesh vertices w.r.t. the handles deformation defined by the animation pose.
30 The mesh is also linked to each handle through a `bindMatrix` that expresses the
31 transformation from the mesh's local frame to the handle local frame.
32 
33 Animations are stored in local space, i.e. each handle transformation is expressed
34 in the handle's frame (transformation from bone's parent for skeleton bones,
35 position in cage's frame for cage points).
36 
37 Hence, in order to deform the character's mesh, one must first use the handle's `bindMatrix`
38 to express the mesh vertices position into the handles' local space before combining the
39 handles local transformations to deform the mesh vertices.
40 
41 A `Ra::Core::Animation::Animation` is a set of key poses, each associated to a key time.
42 When looking for the pose at a specific animation time, the 2 closest surrounding
43 key poses are linearly interpolated to build the querried pose.
44 
45 Though the `Ra::Core::Animation::Animation` class only provides keyframing for poses, which are sets
46 of transformation matrices, the `Ra::Core::Animation::KeyFrame` class, unfinished as of now, provides
47 a more generic interface, usable for any type of animated data.
48 Thus the `Ra::Core::Animation::Animation` class is tagged as deprecated and will be removed once the `Ra::Core::Animation::KeyFrame` class is ready.

50 # The Skeleton-Based Character Animation Plugin
51 
52 The `SkeletonBasedAnimation` plugin (https://gitlab.com/Storm-IRIT/radium-official-plugins/skeletonbasedanimation) provides one plugin specific to skeleton-based character animation and skinning.
53 This plugin provides a `SkeletonComponent`, which manages user interactions with
54 the animation skeleton, enabling posing the character and playing the animations.
55 It also provides a `SkinningComponent`, which is responsible for deforming the
56 object's mesh vertices according to the desired skinning method.

58 # Importing skeleton-based character animation data into Radium
59 
60 In order to import animation related data into Radium, the default loader would be the `Ra::IO::AssimpLoader`,
61  which deals with several standard animation formats (fbx, collada, ...).
62 In case one need to develop his own loader because using a file format not managed by the
63 Assimp library, the Radium Engine provides a set of classes in `Core/Asset` defining data
64 wrappers filled by loaders (see the specific documentation -- which does not exist for now).
65 For now, `Ra::Core::Asset::FileLoaderInterface`s in the Radium Engine produce only one `Ra::Core::Asset::FileData` per loaded file.
66 
67 Regarding character animation data, there are a few things to do in order to ensure correct import by the
68 animation and skinning plugins:
69  * The generated `Ra::Core::Asset::FileData` must contain exactly one `Ra::Core::Asset::HandleData` per animated object, as much `Ra::Core::Asset::AnimationData`
70  as needed per animated object and may contain several or no `Ra::Core::Asset::GeometryData`.
71  Note: for objects composed of several mesh parts, these must be referred by their name when filling the `Ra::Core::Asset::HandleData`.
72  * The animation skeleton, i.e. the bones and hierarchy, must be loaded into the
73  `Ra::Core::Asset::HandleData`, as well as the skinning weights.
74  The bone hierarchy will be defined by the `Ra::Core::Asset::EdgeData`.
75  Each bone will be represented by a `Ra::Core::Asset::HandleComponentData`, storing the bone's name,
76  transform in model space and its per-skinned mesh offset matrix and vertex skinning weights.
77  The offset matrix stores the transformation from mesh space to the node's local space.
78  * Animation keyframes, i.e. couples of time and pose (in local space), if any, must be loaded into an `Ra::Core::Asset::AnimationData`.
79  The whole animation must be stored bone-wise in `Ra::Core::Asset::HandleAnimation`s, whose name must match the bone's.
